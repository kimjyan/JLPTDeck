# JLPTDeck 공격 검토 v3

기준 문서: `RESPONSE_v2.md`  
공격 범위: `[HOLD]`와 `[REFRAME]`만. `[CONCEDE]`, `[ESCALATE]`, `[WITHDRAW]`는 건드리지 않는다. 새 트집 금지. 기존 항목의 방어 논리만 더 판다.

[질문] AVSpeechSynthesizer 표제어 재생을 v1.0 출시 차단에서 제외하는 정확한 기준은 무엇인가?

[약점] 측정만 하고 스케줄에는 안 쓰면 v1.0 사용자는 여전히 찍기 정답을 `.good`으로 저장한다. 측정 인프라는 미래 개선이지 현재 결함 완화가 아니다.

[약점] A/B를 하려면 비교 대상이 있어야 한다. v1.0에서 모든 데이터를 같은 SM-2 처리로 저장하면 "응답 시간 기반 분기"의 사전 기준을 만들기 어렵다.  
[질문] v1.x A/B에서 어떤 그룹, 어떤 지표, 어떤 성공 기준으로 응답 시간 기반 quality를 채택할 것인가?

## 2. 데이터 품질

[심각도] P1  
[지목] `RESPONSE_v2.md` 2.4 / ruby defer  
[작성자 주장] 한자별 매핑 학습은 v1.0 약점으로 기록하고, ruby는 v1.x로 미룬다. v1.0은 한자-읽기 1:1 매핑을 주장하지 않는다.  
[약점] 약점 고지는 제품 손상을 줄이지 않는다. 읽기 필드가 이미 있으므로 최소한 한자어 카드에서 읽기를 먼저 가리고 사용자가 떠올린 뒤 공개하는 방식은 ruby 없이 가능하다.  
[질문] ruby가 없다는 이유로 `headword -> reading` 자가 확인 모드까지 미루는 이유는 무엇인가?

[심각도] P1  
[지목] `RESPONSE_v2.md` 2.5 / 발음 함정 태깅 v1.x  
[작성자 주장] 피치 액센트는 비목표지만 장음·촉음·ん 태깅은 JMdict reading에서 자동 추출 가능하므로 v1.x 후보로 분리한다.  
[약점] 자동 추출 가능하다고 인정했다. 데이터 수집이나 라이선스 문제가 없으면 v1.x로 미루는 이유가 약하다.  
[질문] `reading` 문자열에서 장음·촉음·ん을 탐지해 정답 공개 후 아이콘 하나 붙이는 작업이 v1.0에 못 들어가는 이유는 무엇인가?

## 3. SRS 알고리즘

[심각도] P1  
[지목] `RESPONSE_v2.md` 3.6 / 내부 quality 세분화 v1.x  
[작성자 주장] 응답 시간은 v1.0에 측정하고 quality 분기는 v1.x로 미룬다.  
[약점] 현재도 `SRSQuality`가 존재한다. UI를 바꾸지 않고 `.good`과 `.again` 사이의 내부 처리만 조정할 수 있는데, 이를 v1.x로 미루는 근거가 없다.  
[질문] 응답 시간이 긴 정답을 `.good`이 아니라 더 짧은 interval로 보내는 중간 quality를 v1.0에서 실험하지 않는 이유는 무엇인가?

[심각도] P2  
[지목] `RESPONSE_v2.md` 3.9 / 4지선다 첫 정답 신뢰도 HOLD  
[작성자 주장] 찍기 의심 정의는 응답 시간 분포 데이터가 누적된 뒤 가능하므로 v1.0은 SM-2 표준 interval을 유지한다.  
[약점] 이 항목은 더 밀어붙이지 않는다. 정의 없는 자동 분기는 노이즈라는 주장은 맞다.  
[질문] [WITHDRAW] 응답 시간 분포 없이 찍기 의심 분기를 강제하라는 공격은 철회한다. 대신 v1.0 데이터 수집 후 결정 시점만 문서화하라.

[질문] v1.0에서 시험일을 받지 않는다면 설정 화면에 "20개는 임의 기본값" 외에 어떤 선택 기준을 줄 것인가?
